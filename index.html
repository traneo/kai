The Challenge

AI agents write code.
Teams need confidence.

AI coding agents are exceptionally capable at generating, modifying, and debugging code. But integrating their output into a team-based software delivery process introduces a new set of challenges.

01

No automated validation

The agent's output — lint errors, type failures, broken tests, security issues — goes unchecked unless a human manually reviews every line.

→

Kai runs configurable validation gates after every step: lint, typecheck, tests, diff review, security scan, and custom plugins.

02

Blind trust required

There's no record of what the AI did, what decisions it made, or what changed from one iteration to the next. Debugging issues means asking the agent to re-explain itself.

→

Every prompt, file change, approval, and result is recorded in an immutable audit trail with 14 event types, queryable via API, CLI, or UI.

03

No human approval gates

Either the AI pushes directly to production, or a human must sit and watch every step. There's no middle ground for "review before merge."

→

Steps can require human approval before proceeding. Reviewers see the diff, leave feedback, approve, reject, or request changes — all from the pipeline UI or CLI.

04

Disconnected from git workflow

AI changes land outside the team's existing PR workflow. Branching, committing, reviewing, and merging are manual afterthoughts.

→

Full git lifecycle automation: clone, branch, commit, push, and PR creation. Supports GitHub, GitLab, Bitbucket, and Forgejo.

05

No cross-team observability

Each developer's AI usage is an island. There's no centralized view of what agents are doing, how they're performing, or where they're failing across the organization.

→

Centralized log ingestion from all services with real-time SSE streaming. Query by service, level, run ID, or step ID. PostgreSQL or in-memory storage.

06

Vendor lock-in

Committing to one AI provider or one coding agent means your entire workflow depends on their roadmap, pricing, and availability.

→

Pluggable runner architecture: kai-code (C# .NET), opencode, and Claude Code are first-class. Swap backends per pipeline or per step.

The common thread: governance

Kai doesn't replace AI coding agents. It adds the CI/CD‑inspired layer that production teams need: validation, audit, approvals, git automation, and observability — wrapped around any AI coding tool, in a declarative YAML pipeline format that lives in your repository.

How It Works

AI agents do the work.
Kai runs the process.

Define your pipeline as a YAML file. Kai takes care of execution, validation, and delivery — while keeping humans in the loop where it matters.

1

Define

A pipeline.yaml declares the project, repository, output mode, and a DAG of steps — each with a prompt, validation gates, approval policy, and per-step LLM configuration.

2

Submit

Submit via kaictl pipeline create, the Platform UI, or the REST API. The orchestrator parses, validates, and resolves the DAG.

3

Dispatch

Ready steps are assigned to idle agents via gRPC. Each agent creates a sandboxed workspace, clones the repo, and invokes its runner plugin (kai-code, opencode, or Claude Code).

4

Execute

The runner executes the step prompt against its LLM — reading files, writing code, running commands. Logs and file changes stream back in real time via the MissionEvent stream.

5

Validate

On completion, the orchestrator runs the configured validation gates: exit_zero, lint, typecheck, tests, diff_review, security_scan, license_check, and custom plugin gates.

6

Approve

If the step requires approval, it blocks and presents the diff. A human reviews and approves, rejects, or requests changes via the UI, CLI, or API.

7

Deliver

After all steps pass: git commit, push the feature branch, and create a pull request. Every event is recorded in the audit trail and streamed to observability.

✓ Declarative YAML pipelines — version-controlled alongside your code
✓ Parallel DAG execution — steps run concurrently where dependencies allow
✓ Retry with backoff — linear or exponential, configurable per step
✓ Hot-reload config — pool config, auth, backends update without restart

Capabilities

What you get out of the box.

✅

Validation Gates

Lint, typecheck, tests, diff review, security scan, license check, breaking change detection, and code quality gates. Pluggable for custom checks.

👁️

Audit Trail

14 event types covering pipeline lifecycle, step execution, agent assignments, approvals, and system events. Queryable via REST API, CLI, or UI.

✋

Human Approvals

Block steps until human sign-off. Review diffs, leave feedback, approve, reject, or request changes. Works from the UI, CLI, or REST API.

🔗

Git Integration

Automated clone, branch, commit, push, and PR creation. GitHub, GitLab, Bitbucket, and Forgejo supported via pluggable git provider system.

🔀

DAG Scheduling

Static dependency analysis with cycle detection. Steps with no interdependencies execute in parallel. Configurable retry with linear or exponential backoff.

📊

Observability

Centralized log ingestion from all services. SSE live stream, per-service filtering, and PostgreSQL or in-memory storage with file dump for debugging.

🔌

Pluggable Runners

kai-code (C# .NET), opencode, Claude Code. Swap per step, per pipeline, or per pool. Custom runners are a small Go binary via the plugin system.

📝

Plan Builder

Describe your project in plain English. A conversational LLM agent builds the pipeline YAML for you — no need to learn the schema from scratch.
